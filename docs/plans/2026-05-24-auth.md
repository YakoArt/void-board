# План реализации: Аутентификация

Дизайн: `docs/designs/2026-05-19-auth.md`

---

## Обзор

Реализация полного цикла аутентификации: регистрация, вход, выход, JWT-сессии с ротацией refresh-токенов, защищённые маршруты, страница профиля.

Фаза 0 закладывает все Prisma-модели проекта одной миграцией (User, RefreshToken, Project, Issue, Label, Comment). Остальные фазы реализуют только auth-функциональность.

**Рекомендуемый размер команды:** 3 агента

---

## Фаза 0 — Модель данных

*Без зависимостей. Одна задача, выполняется первой.*

Все сущности проекта описываются в Prisma-схеме одной миграцией. Это фундамент для всех будущих модулей.

### Задача 0.1: Prisma-схема всех сущностей

Описать полную модель данных Voidboard в `schema.prisma`, сгенерировать клиент, создать миграцию.

- [ ] Обновить `packages/database/prisma/schema.prisma` — добавить модель `User`:
  - `id` (uuid, `@db.Uuid`), `email` (unique), `passwordHash` (`@map("password_hash")`), `name`, `avatarUrl?` (`@map("avatar_url")`), `createdAt`, `updatedAt`
  - Связи: `refreshTokens`, `ownedProjects`, `reportedIssues`, `assignedIssues`, `comments`
  - `@@map("users")`
- [ ] Добавить модель `RefreshToken`:
  - `id` (uuid), `token` (unique), `userId` (`@map("user_id")`, `@db.Uuid`), `expiresAt`, `createdAt`
  - Связь: `user` с `onDelete: Cascade`
  - Индексы: `userId`, `expiresAt`
  - `@@map("refresh_tokens")`
- [ ] Добавить модель `Project`:
  - `id` (uuid, `@db.Uuid`), `slug` (unique), `name`, `description?`, `ownerId` (`@map("owner_id")`, `@db.Uuid`), `createdAt`, `updatedAt`
  - Связи: `owner` (User), `issues`, `labels`
  - Индекс: `ownerId`

  - Индексы: `projectId`, `reporterId`, `assigneeId`, `status`
  - `@@map("issues")`
- [ ] Добавить enum `IssueStatus`: `backlog`, `todo`, `in_progress`, `in_review`, `done`

  - `id` (uuid, `@db.Uuid`), `name`, `color` (`@db.VarChar(7)` — hex-цвет), `projectId` (`@map("project_id")`, `@db.Uuid`), `createdAt`
  - Связи: `project` (Project, onDelete: Cascade), `issues` (Issue[])
  - Уникальный составной индекс: `@@unique([projectId, name])` (уникальность имени в рамках проекта)
  - `@@map("labels")`
- [ ] Добавить модель `Comment`:
  - `id` (uuid, `@db.Uuid`), `body` (text), `authorId` (`@map("author_id")`, `@db.Uuid`), `issueId` (`@map("issue_id")`, `@db.Uuid`), `createdAt`, `updatedAt`
  - Связи: `author` (User), `issue` (Issue, onDelete: Cascade)
  - Индексы: `issueId`, `authorId`
  - `@@map("comments")`
- [ ] Добавить неявную M:N связь Issue ↔ Label (Prisma implicit many-to-many через `_IssueLabels`)
- [ ] Запустить `prisma migrate dev --name init-all-entities` для создания миграции
- [ ] Запустить `prisma generate` для генерации клиента
- [ ] Проверить что миграция применилась без ошибок

**Файлы:**
- `packages/database/prisma/schema.prisma`
- `packages/database/prisma/migrations/` (генерируется)

**Ключевые решения:**
- `uuid()` + `@db.Uuid` для всех ID — нативный PostgreSQL uuid, 16 байт
- `@@map` для snake_case в БД при camelCase в коде
- `onDelete: Cascade` на Issue→Project, Label→Project, Comment→Issue, RefreshToken→User
- Enum'ы `IssueStatus` и `IssuePriority` — нативные PostgreSQL enum
- `Label.name` уникален в рамках Project (составной `@@unique`)
- `aiSuggested` флаг на Issue для пометки AI-сгенерированных acceptance_criteria

## Фаза 2 — Бэкенд-модули + Фронтенд-страницы

*Зависит от: Фазы 1 (нужны PrismaModule, ConfigModule, зависимости)*

Два независимых бэкенд-модуля + фронтенд-страницы могут работать параллельно.

### Задача 2.1: AuthModule (бэкенд)

Реализовать полный модуль аутентификации: JWT-стратегия, guards, декораторы, DTO, сервис, контроллер.

- [ ] Создать `apps/api/src/auth/dto/register.dto.ts` — `email` (IsEmail), `password` (MinLength 8), `name` (IsString, IsNotEmpty)
- [ ] Создать `apps/api/src/auth/dto/login.dto.ts` — `email` (IsEmail), `password` (IsString)
- [ ] Создать `apps/api/src/auth/dto/auth-response.dto.ts` — интерфейс ответа `{ accessToken, user }`
- [ ] Создать `apps/api/src/auth/decorators/public.decorator.ts` — `@Public()` через `SetMetadata`
- [ ] Создать `apps/api/src/auth/decorators/current-user.decorator.ts` — `@CurrentUser()` через `createParamDecorator`
- [ ] Создать `apps/api/src/auth/strategies/jwt.strategy.ts` — Passport JWT strategy, извлечение токена из Authorization header, валидация payload, возврат `{ id, email }`
- [ ] Создать `apps/api/src/auth/guards/jwt-auth.guard.ts` — `AuthGuard('jwt')` с поддержкой `@Public()` через Reflector
- [ ] Создать `apps/api/src/auth/auth.service.ts` — бизнес-логика:
  - `register()` — проверка уникальности email, хеширование пароля (bcryptjs, 10 rounds), создание пользователя, генерация токенов
  - `login()` — поиск по email, timing-safe сравнение (всегда считать bcrypt даже если email не найден), генерация токенов
  - `refresh()` — валидация refresh-токена (sha256 хеш для сравнения с БД), ротация (удалить старый, создать новый), генерация нового access-токена
  - `logout()` — удаление refresh-токена из БД
  - `generateTokens()` — приватный метод, создаёт access (JWT, 15 мин) и refresh (crypto.randomUUID, 7 дней), сохраняет хеш refresh в БД
  - `invalidateAllUserTokens()` — удаление всех refresh-токенов пользователя (при обнаружении повторного использования)
- [ ] Создать `apps/api/src/auth/auth.controller.ts` — эндпоинты:
  - `POST /auth/register` — `@Public()`, ставит refresh cookie, возвращает `{ accessToken, user }`, статус 201
  - `POST /auth/login` — `@Public()`, ставит refresh cookie, возвращает `{ accessToken, user }`
  - `POST /auth/refresh` — `@Public()`, читает cookie, ротирует, возвращает `{ accessToken }`
  - `POST /auth/logout` — читает cookie, удаляет токен, чистит cookie, статус 204
  - Cookie-опции: `httpOnly: true`, `secure` из env, `sameSite: 'strict'`, `path: '/api/auth'`, `maxAge` из env
- [ ] Создать `apps/api/src/auth/auth.module.ts` — импорт `JwtModule.registerAsync()`, `PassportModule`, экспорт `JwtService`
- [ ] Обновить `apps/api/src/app/app.module.ts` — импортировать `AuthModule`, `PrismaModule`, зарегистрировать `JwtAuthGuard` как `APP_GUARD`, зарегистрировать `ThrottlerGuard` как `APP_GUARD`
- [ ] Применить `@Throttle()` с лимитами из дизайна: 5/мин на `/login`, 3/мин на `/register`

**Файлы:**
- `apps/api/src/auth/` (вся директория — новая)
- `apps/api/src/app/app.module.ts` (добавление импортов AuthModule, PrismaModule, APP_GUARD)

## Модель данных (справка)

Полная карта связей для задачи 0.1:

```
User    1 —— N   RefreshToken  (cascade delete)
User    1 —— N   Project       (owner)
User    1 —— N   Issue         (reporter)
User    1 —— N   Issue         (assignee, optional)
User    1 —— N   Comment       (author)
Project 1 —— N   Issue         (cascade delete)
Project 1 —— N   Label         (cascade delete)
Issue   M —— N   Label         (implicit many-to-many)
Issue   1 —— N   Comment       (cascade delete)
```

Enum'ы:
- `IssueStatus`: `backlog` | `todo` | `in_progress` | `in_review` | `done`
- `IssuePriority`: `low` | `medium` | `high` | `urgent`

Инварианты (для будущих модулей, не для auth):
- Issue создаётся в `backlog`, priority = `medium`, без assignee
- `reporter` заполняется текущим пользователем, не меняется
- `closedAt` ставится при переходе в `done`, обнуляется при выходе
- `Label.name` уникален в рамках Project (без учёта регистра)
- Comment может редактировать/удалять только автор
