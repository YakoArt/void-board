# Дизайн: Аутентификация (Auth)

> **Дата:** 2026-05-19
> **Статус:** Согласован
> **Этап:** 1 — первый feature-модуль Voidboard

---

## Цель

Реализовать полный цикл аутентификации: регистрация по email/паролю, вход, выход, JWT-сессии с ротацией refresh-токенов, защищённые маршруты, страница профиля. Без OAuth, без email-верификации — это MVP.

---

## Решения

| Вопрос | Решение |
|---|---|
| JWT-стратегия | Access-токен (15 мин) + refresh-токен (7 дней) с ротацией |
| Хранение на клиенте | Refresh — httpOnly cookie; access — в памяти (JS-переменная) |
| Email-верификация | Нет (MVP) |
| UI-скоуп | Login, Register, Logout + страница профиля (имя, аватар, смена пароля) |

## Модель данных

```prisma
model User {
  id            String    @id @default(uuid()) @db.Uuid
  email         String    @unique
  passwordHash  String    @map("password_hash")
  name          String
  avatarUrl     String?   @map("avatar_url")
  createdAt     DateTime  @default(now()) @map("created_at")
  updatedAt     DateTime  @updatedAt @map("updated_at")

  refreshTokens RefreshToken[]

  @@map("users")
}

model RefreshToken {
  id        String   @id @default(uuid()) @db.Uuid
  token     String   @unique
  userId    String   @map("user_id") @db.Uuid
  expiresAt DateTime @map("expires_at")
  createdAt DateTime @default(now()) @map("created_at")

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@index([expiresAt])
  @@map("refresh_tokens")
}
```

**Ключевые решения:**
- `uuid()` для ID — стандартный UUID v4, маппится на нативный PostgreSQL-тип `uuid` (`@db.Uuid`) — 16 байт вместо 36 байт строки
- Refresh-токены в отдельной таблице — один пользователь может иметь несколько активных сессий (разные устройства)
- `onDelete: Cascade` — удаление пользователя чистит все его токены
- `@@map` для snake_case в БД при camelCase в коде
- Индекс по `expiresAt` — для периодической чистки просроченных токенов

## API-эндпоинты

### Auth

| Эндпоинт | Body | Ответ (успех) | Cookie |
|---|---|---|---|
| `POST /api/auth/register` | `{ email, password, name }` | `201` — `{ accessToken, user }` | Ставит `refresh_token` httpOnly |
| `POST /api/auth/login` | `{ email, password }` | `200` — `{ accessToken, user }` | Ставит `refresh_token` httpOnly |
| `POST /api/auth/refresh` | — (cookie) | `200` — `{ accessToken }` | Ротирует `refresh_token` cookie |
| `POST /api/auth/logout` | — (cookie) | `204` | Удаляет `refresh_token` cookie |

### Users

| Эндпоинт | Body | Ответ (успех) |
|---|---|---|
| `GET /api/users/me` | — | `200` — `{ id, email, name, avatarUrl, createdAt }` |
| `PATCH /api/users/me` | `{ name?, avatarUrl? }` | `200` — обновлённый user |
| `PATCH /api/users/me/password` | `{ currentPassword, newPassword }` | `204` |

**Ключевые решения:**
- Регистрация сразу логинит — возвращает токены, не нужен отдельный шаг
- Refresh через cookie — фронтенд не трогает refresh-токен руками
- Ротация refresh-токена при каждом `/refresh` — старый удаляется, создаётся новый
- `password` / `passwordHash` нигде не возвращается в ответах

## Бэкенд-архитектура (NestJS)

```
apps/api/src/
├── main.ts
├── app.module.ts              — корневой, импортирует все модули
│
├── auth/
│   ├── auth.module.ts         — AuthModule
│   ├── auth.controller.ts     — register, login, refresh, logout
│   ├── auth.service.ts        — бизнес-логика (хеширование, токены, ротация)
│   ├── strategies/
│   │   └── jwt.strategy.ts    — Passport JWT strategy (из Authorization header)
│   ├── guards/
│   │   └── jwt-auth.guard.ts  — AuthGuard('jwt'), глобальный через APP_GUARD
│   ├── decorators/
│   │   ├── current-user.decorator.ts  — @CurrentUser() → user из request
│   │   └── public.decorator.ts        — @Public() → эндпоинт без авторизации
│   └── dto/
│       ├── register.dto.ts
│       ├── login.dto.ts
│       └── auth-response.dto.ts
│
├── users/
│   ├── users.module.ts        — UsersModule
│   ├── users.controller.ts    — me, updateProfile, changePassword
│   ├── users.service.ts       — CRUD по User
│   └── dto/
│       ├── user-response.dto.ts
│       ├── update-profile.dto.ts
│       └── change-password.dto.ts
│
├── prisma/
│   ├── prisma.module.ts       — PrismaModule (Global)
│   └── prisma.service.ts      — extends PrismaClient, onModuleInit connect
│
└── common/
    ├── filters/
    │   └── http-exception.filter.ts   — единый формат ошибок
    └── pipes/
        └── validation.pipe.ts         — глобальный ValidationPipe
```

**Ключевые решения:**
- **JWT Guard глобально** через `APP_GUARD`. Все эндпоинты защищены по умолчанию. Открытые помечаются `@Public()` (SetMetadata + Reflector). Безопаснее, чем навешивать guard вручную.
- **PrismaModule глобальный** — один инстанс PrismaClient на всё приложение.
- **Passport + @nestjs/jwt** — стандартная связка для NestJS.
- **Валидация через class-validator** — DTO с декораторами, глобальный `ValidationPipe` с `whitelist: true` и `transform: true`.
- **Хеширование — bcryptjs** (чистый JS, без native-зависимостей). Salt rounds = 10.

## Фронтенд-архитектура (Angular)

```
apps/web/src/app/
├── app.ts                        — корневой компонент (shell)
├── app.routes.ts                 — маршруты верхнего уровня
├── app.config.ts                 — провайдеры (router, primeng, http)
│
├── core/
│   ├── auth/
│   │   ├── auth.service.ts           — login, register, logout, refresh, состояние
│   │   ├── auth.interceptor.ts       — Authorization header + auto-refresh при 401
│   │   ├── auth.guard.ts             — canActivate для защищённых маршрутов
│   │   ├── guest.guard.ts            — canActivate для login/register (редирект если залогинен)
│   │   └── auth.models.ts            — интерфейсы (User, AuthResponse, LoginRequest и т.д.)
│   └── api/
│       └── api.config.ts            — базовый URL бэкенда
│
├── features/
│   ├── auth/
│   │   ├── login/
│   │   │   └── login.component.ts    — standalone, страница логина
│   │   └── register/
│   │       └── register.component.ts — standalone, страница регистрации
│   └── profile/
│       └── profile.component.ts      — standalone, просмотр/редактирование профиля
│
└── shared/
    └── layout/
        └── header/
            └── header.component.ts   — лого, имя пользователя, logout
```

**Ключевые решения:**
- **Signals** — `AuthService` хранит состояние через `signal<User | null>`. Computed-сигналы `isAuthenticated`, `currentUser`. Никаких BehaviorSubject.
- **Functional HTTP interceptor** — подставляет `Authorization: Bearer {accessToken}`. При 401 — автоматический `/auth/refresh`, повтор запроса. Если refresh тоже 401 — logout, редирект на `/login`.
- **Два guard'а**: `authGuard` (защита приватных маршрутов) и `guestGuard` (редирект залогиненных с login/register).
- **Standalone-компоненты** — никаких NgModules для фич. Lazy loading через `loadComponent`.
- **Восстановление сессии при перезагрузке** — при старте `AuthService` вызывает `POST /auth/refresh` (cookie уйдёт автоматически). Успех — залогинен, неудача — гость.

### Маршруты

```
/login          — LoginComponent (guestGuard)
/register       — RegisterComponent (guestGuard)
/profile        — ProfileComponent (authGuard)
/               — редирект на /login (пока нет dashboard)
```

## Безопасность

| Мера | Реализация |
|---|---|
| Хеширование паролей | bcryptjs, salt rounds = 10 |
| Refresh-токен в БД | Хранится хешированным (sha256). В cookie — оригинал, в БД — хеш. Утечка БД не компрометирует сессии |
| httpOnly cookie | `httpOnly: true`, `secure: true` (prod), `sameSite: 'strict'`, `path: '/api/auth'` |
| Ротация refresh-токена | При каждом `/refresh` старый удаляется, создаётся новый. Повторное использование старого — инвалидация всех токенов пользователя |
| Валидация паролей | Минимум 8 символов |
| Timing-safe | Email не найден → всё равно считаем bcrypt-хеш, чтобы время ответа не выдавало существование аккаунта |
| Rate limiting | `@nestjs/throttler` — 5 попыток/мин на `/login`, 3/мин на `/register` |
| CORS | Конкретный origin фронтенда, `credentials: true` |

### Коды ошибок

| Ситуация | HTTP-код |
|---|---|
| Невалидный body (DTO) | `400 Bad Request` |
| Неверный email или пароль | `401 Unauthorized` |
| Невалидный/просроченный access-токен | `401 Unauthorized` |
| Невалидный/просроченный refresh-токен | `401 Unauthorized` |
| Email уже занят | `409 Conflict` |
| Неверный текущий пароль (смена) | `403 Forbidden` |
| Rate limit | `429 Too Many Requests` |

### Единый формат ошибок

```json
{
  "statusCode": 409,
  "error": "Conflict",
  "message": "Пользователь с таким email уже существует"
}
```

## Конфигурация

### Переменные окружения (бэкенд)

```env
# База данных (уже есть)
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/voidboard?schema=public"

# JWT
JWT_ACCESS_SECRET="random-secret-here"
JWT_ACCESS_TTL="15m"
JWT_REFRESH_TTL="7d"

# Cookie
COOKIE_SECURE="false"
COOKIE_DOMAIN="localhost"

# Rate limiting
THROTTLE_TTL="60000"
THROTTLE_LIMIT="10"

# CORS
CORS_ORIGIN="http://localhost:4200"

# Общее
NODE_ENV="development"
PORT="3000"
```

Загрузка через `@nestjs/config` с валидацией — приложение падает при старте, если обязательные переменные не заданы.

```typescript
export const environment = {
  apiUrl: 'http://localhost:3000/api',
```
