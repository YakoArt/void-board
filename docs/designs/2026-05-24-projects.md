# Дизайн: Projects — базовый CRUD

> Дата: 2026-05-24
> Статус: Согласован
> Автор: Claude + Саша

---

## Цель

Реализовать полный CRUD проектов: создание, просмотр, редактирование, удаление. Текущий пользователь автоматически становится owner. Навигация: список проектов → страница проекта. Это первый шаг после аутентификации — основа для всех последующих фич (Issues, AI-триаж и т.д.).

---

## Принятые решения

| Вопрос | Решение | Обоснование |
|---|---|---|
| Scope | Полный CRUD (Create, Read, Update, Delete) | Если можно создать — должно быть можно и поправить |
| Slug-генерация | Автогенерация из name + случайный суффикс (4 символа) всегда | Безопасно (не раскрывает существование чужих проектов), предсказуемо, без edge cases |
| Доступ к проектам | Только свои (owner) | Простая изоляция для MVP, коллаборацию добавим позже |
| Удаление с issues | Каскадное удаление + диалог подтверждения на фронте | Cascade уже в Prisma-схеме, фронт предупреждает пользователя |
| Главная страница | `/projects` (вместо `/profile`) | Трекер — это проекты, профиль доступен через хедер |
| Страница проекта | Шапка + плейсхолдер для issues | Готовим место для следующего этапа |

## Модель данных

Используем существующую Prisma-схему **без изменений**:

```prisma
model Project {
  id          String   @id @default(uuid()) @map("id") @db.Uuid
  slug        String   @unique
  name        String
  description String?
  ownerId     String   @map("owner_id") @db.Uuid
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  owner  User    @relation(fields: [ownerId], references: [id])
  issues Issue[]
  labels Label[]

  @@index([ownerId])
  @@map("projects")
}
```

## API

Все эндпоинты защищены `JwtAuthGuard` (глобальный). Префикс: `/api/projects`.

### Эндпоинты

| Метод | Путь | Описание | Тело запроса | Ответ |
|---|---|---|---|---|
| `POST` | `/api/projects` | Создать проект | `CreateProjectDto` | `201` + `ProjectResponseDto` |
| `GET` | `/api/projects` | Список проектов текущего пользователя | — | `200` + `ProjectResponseDto[]` |
| `GET` | `/api/projects/:slug` | Получить проект по slug | — | `200` + `ProjectResponseDto` |
| `PATCH` | `/api/projects/:slug` | Обновить проект | `UpdateProjectDto` | `200` + `ProjectResponseDto` |
| `DELETE` | `/api/projects/:slug` | Удалить проект | — | `204` No Content |

### DTOs

**CreateProjectDto:**
```typescript
interface CreateProjectDto {
  name: string;        // required, 1–100 символов
  description?: string; // optional, max 2000 символов
}
```

**UpdateProjectDto:**
```typescript
interface UpdateProjectDto {
  name?: string;        // 1–100 символов
  description?: string | null; // max 2000; null — очистить поле
}
```

**ProjectResponseDto:**
```typescript
interface ProjectResponseDto {
  id: string;
  slug: string;
  name: string;
  description: string | null;
  createdAt: string; // ISO 8601
  updatedAt: string; // ISO 8601
}
```

> `ownerId` не включён в ответ — пользователь видит только свои проекты, поле избыточно.

### Slug-генерация

Алгоритм (на бэкенде):

1. `name` → транслитерация → lowercase → замена пробелов/спецсимволов на `-` → обрезка до 50 символов
2. Генерация случайного суффикса: 4 символа (a-z, 0-9)
3. Итоговый slug: `{transliterated-name}-{suffix}` (например: `my-app-x7k2`)
4. Проверка уникальности в БД. При коллизии (крайне маловероятно) — перегенерация суффикса (до 5 попыток, потом 500)
5. При обновлении `name` — slug **не меняется** (стабильные URL)
6. Если name состоит только из спецсимволов и транслитерация даёт пустую строку → `400 Bad Request` с сообщением «Название должно содержать хотя бы одну букву или цифру»

Библиотека: `slugify` — маленькая, проверенная, поддерживает транслитерацию.

### Авторизация

- `GET /projects` — фильтр `WHERE ownerId = currentUser.id`
- `GET/PATCH/DELETE /projects/:slug` — проверка `ownerId === currentUser.id`
- Если проект не найден **или** принадлежит другому пользователю → `404 Project not found` (не 403, чтобы не раскрывать существование чужих проектов)

## Фронтенд

### Маршруты

| Путь | Компонент | Guard | Загрузка |
|---|---|---|---|
| `/projects` | `ProjectListComponent` | `authGuard` | lazy `loadComponent` |
| `/projects/new` | `ProjectCreateComponent` | `authGuard` | lazy `loadComponent` |
| `/projects/:slug` | `ProjectDetailComponent` | `authGuard` | lazy `loadComponent` |
| `/projects/:slug/edit` | `ProjectEditComponent` | `authGuard` | lazy `loadComponent` |

> **Порядок маршрутов важен:** `/projects/new` должен быть **перед** `/projects/:slug`, иначе `new` будет интерпретирован как slug.

**Изменения в существующих маршрутах:**
- `/` → redirect на `/projects` (было `/profile`)
- `**` → redirect на `/projects` (было `/profile`)
- `guestGuard` → redirect на `/projects` (было `/profile`)
- Login/Register → после успеха навигация на `/projects`

### Компоненты

**1. `ProjectListComponent`** (`features/projects/project-list/`)
- Список карточек проектов текущего пользователя
- Каждая карточка: name, description (обрезанное до ~100 символов), дата создания
- Клик по карточке → `/projects/:slug`
- Кнопка «Создать проект» → `/projects/new`
- Empty state (нет проектов): текст + CTA «Создать первый проект»

**2. `ProjectCreateComponent`** (`features/projects/project-create/`)
- Reactive form: name (required), description (optional, textarea)
- Валидация: name 1–100 символов, description max 2000
- После успеха → навигация на `/projects/:slug` (slug из ответа сервера)
- При ошибке → сообщение под формой

**3. `ProjectDetailComponent`** (`features/projects/project-detail/`)
- Шапка: name, description, дата создания
- Кнопки действий: «Редактировать» → `/projects/:slug/edit`, «Удалить»
- Удаление: PrimeNG `ConfirmDialog` с предупреждением о каскадном удалении → после успеха → `/projects`
- Область контента: empty state «Пока нет issues» (плейсхолдер для следующего этапа)

**4. `ProjectEditComponent`** (`features/projects/project-edit/`)
- Та же форма что и создание, предзаполненная данными проекта
- Загрузка данных проекта по slug из URL
- После успеха → навигация на `/projects/:slug`

### Сервис

**`ProjectsService`** (`core/projects/projects.service.ts`):
- `getProjects(): Observable<ProjectResponseDto[]>`
- `getProject(slug: string): Observable<ProjectResponseDto>`
- `createProject(dto: CreateProjectDto): Observable<ProjectResponseDto>`
- `updateProject(slug: string, dto: UpdateProjectDto): Observable<ProjectResponseDto>`
- `deleteProject(slug: string): Observable<void>`

### Хедер

- Добавить ссылку «Проекты» в навигацию (для авторизованных пользователей)

## Обработка ошибок

### Бэкенд

| Ситуация | Ответ |
|---|---|
| Slug-коллизия при создании | Перегенерация суффикса (до 5 попыток, потом 500) |
| Проект не найден / чужой проект | `404 Project not found` |
| Name из одних спецсимволов | `400 Bad Request` — «Название должно содержать хотя бы одну букву или цифру» |
| Дублирование name у одного пользователя | Разрешено — slug уникален глобально |
| Пустое description при обновлении | `null` — очищает поле; отсутствие поля — не трогает |
| Удаление проекта с issues | Каскадное удаление (Prisma cascade) |

### Фронтенд

| Ситуация | Поведение |
|---|---|
| Ошибка загрузки списка | Сообщение об ошибке + кнопка «Повторить» |
| 404 при открытии проекта | Redirect на `/projects` + toast «Проект не найден» |
| Ошибка при создании/обновлении | Сообщение об ошибке под формой (паттерн из auth) |
| Ошибка при удалении | Toast с ошибкой, диалог закрывается |
| Пустой список проектов | Empty state с CTA «Создать первый проект» |

### Валидация (дублируется на фронте и бэке)

| Поле | Правила |
|---|---|
| `name` | Required, 1–100 символов |
| `description` | Optional, max 2000 символов |

## Вне scope (YAGNI)

- Коллаборация / ProjectMember — будет отдельной фичей
- Пагинация списка проектов — не нужна для MVP (десятки проектов, не тысячи)
- Поиск/фильтрация проектов — аналогично
- Архивирование проектов (soft delete) — усложнение без пользы на этом этапе
- Аватар/иконка проекта — косметика, не MVP
- Сортировка списка — по умолчанию по `createdAt DESC`, этого достаточно
