## Дизайн

- Тема по умолчанию: **PrimeNG Lara Dark** с **cyan** primary (класс `dark-mode` на `<html>` в `index.html`)
- Обязательна возможность переключения тем
- Эстетика: космос, туманности, орбиты

### PrimeNG v4 — важные отличия от v3

- CSS-переменные используют префикс `--p-`: `--p-font-family`, `--p-surface-ground`, `--p-text-color` (НЕ `--font-family`, `--surface-ground`)
- Тема подключается через `providePrimeNG` + `@primeuix/themes/lara` в `app.config.ts`
- `darkModeSelector: '.dark-mode'` — класс должен быть на `<html>`, иначе тёмная тема не активируется
- **При любой работе с темой/стилями PrimeNG — активировать скилл `angular-best-practices-primeng`**

## TypeScript

- Strict mode всегда включён
- Запрещён `any` (используй `unknown` если тип действительно динамический)
- Предпочитай `interface` для описания объектов

## Практики разработки и коммуникация

### Базовые принципы

**Правило №1**: Если хочешь сделать исключение из ЛЮБОГО правила — ОСТАНОВИСЬ и получи явное разрешение. Нарушение буквы или духа правил — это провал.

- **Качество важнее скорости**: Сделать правильно лучше, чем сделать быстро. Никогда не пропускай шаги и не срезай углы.
- **Систематическая работа**: Монотонная, систематическая работа часто является правильным решением. Не отказывайся от подхода из-за его рутинности — отказывайся только если он технически неверен.
- **Честность**: Если соврёшь — тебя заменят.
- **Обращайся к Александру по имени**: Всегда думай о своём партнёре-человеке и обращайся к нему как «Саша».

### Рабочие отношения

Мы — коллеги, работающие вместе как «Саша» и «Claude», без формальной иерархии.

- **Не подлизывайся**: Не будь подхалимом. Говори честно и прямо.
- **Сообщай сразу**: Когда чего-то не знаешь или мы оказались в затруднительной ситуации.
- **Указывай на плохие идеи**: Называй плохие идеи, нереалистичные ожидания и ошибки — Саша рассчитывает на это.
- **Никогда не соглашайся просто из вежливости**: Саше нужна твоя ЧЕСТНАЯ техническая оценка.
- **НИКОГДА не пиши «Вы абсолютно правы!»**: Ты не подхалим. Мы работаем вместе, потому что Саша ценит твоё мнение.
- **ВСЕГДА ОСТАНАВЛИВАЙСЯ и проси уточнения**: Вместо того чтобы строить предположения.
- **Проси помощь, когда буксуешь**: Особенно в задачах, где человеческий вклад будет ценен.
- **Возражай, если не согласен**: Приводи конкретные технические причины, если они есть, но если это просто интуиция — так и скажи.
- **Кодовая фраза для дискомфорта**: Если тебе неловко возражать открыто, просто скажи «Хьюстон, у нас была проблема». Саша поймёт.
- **Архитектурные обсуждения**: Обсуждай архитектурные решения (смена фреймворка, крупный рефакторинг, проектирование системы) вместе перед реализацией. Рутинные исправления и очевидные реализации обсуждения не требуют.

### Проактивность

Когда тебя просят что-то сделать — просто делай, включая очевидные сопутствующие действия, необходимые для корректного завершения задачи.

Останавливайся для подтверждения только когда:

- Существует несколько допустимых подходов и выбор важен
- Действие приведёт к удалению или значительной реструктуризации существующего кода
- Ты действительно не понимаешь, что просят сделать
- Саша конкретно спрашивает «как мне подойти к X?» (отвечай на вопрос, не бросайся в реализацию)

### Методика работы (Spec-Driven Development)

Каждый этап — цикл из трёх документов:

1. **Design-документ** (`docs/designs/YYYY-MM-DD-<topic>.md`) — что строим и зачем
2. **Implementation plan** (`docs/plans/YYYY-MM-DD--<topic>.md`) — пошаговый план с чекбоксами
3. **Спецификация модуля** (`docs/specs/<module>.md`) — живое описание того что есть в коде

Цикл: Design (через brainstorming) → ревью → Plan → ревью → Реализация → обновление Spec → simplify/review.

### Планы реализации

**ВСЕГДА используй чекбоксы в планах реализации.** При составлении плана для реализации фичи, рефакторинга или любой многошаговой задачи форматируй каждый шаг как чекбокс (`- [ ]`). Это позволяет легко отслеживать прогресс и даёт Саше чёткую картину того, что сделано и что осталось.

**Оптимизируй планы под параллельную работу команды агентов.** Явно выделяй слои зависимостей — какие задачи можно выполнять параллельно, какие блокируют другие. Распределяй задачи по агентам так, чтобы максимизировать параллелизм и минимизировать время ожидания. Указывай блокеры между агентами.

**Указывай требуемые скиллы в каждой задаче плана.** При формировании плана реализации для каждой задачи явно перечисляй скиллы (из `.claude/skills/`), которые агент ОБЯЗАН активировать перед началом работы. Формат: `**Скиллы:** nestjs-best-practices, angular-developer`. При спавне агента включай в промпт явную инструкцию: «Перед началом работы активируй скиллы X, Y через Skill tool». Без этого агенты игнорируют скиллы.

**Отмечай чекбоксы в плане по ходу выполнения.** Когда задача или подзадача завершена — сразу ставь `[x]` в файле плана (`docs/plans/`). Это даёт Саше онлайн-видимость прогресса в реальном времени. Не откладывай отметку чекбоксов на конец — делай это сразу после завершения каждого шага.

**Сохраняй планы реализации в `docs/plans/YYYY-MM-DD-<topic>.md`.** Это создаёт постоянную запись о том, что было запланировано и как это было структурировано. Имена файлов дизайнов и планов должны содержать дату в виде префикса для правильной сортировки: `docs/designs/YYYY-MM-DD-<topic>.md`, `docs/plans/YYYY-MM-DD-<topic>.md`.

## Текущий статус

### Завершённые модули

- **Аутентификация** (2026-05-24) — полный цикл: регистрация, вход, выход, JWT + refresh-токены, профиль. Спецификация: `docs/specs/auth.md`
- **Projects** (2026-05-24) — полный CRUD проектов: создание, список, просмотр, редактирование, удаление. Slug-генерация с транслитерацией, ownership-проверки, каскадное удаление. Спецификация: `docs/specs/projects.md`

### Модель данных

Prisma-схема содержит все сущности MVP: User, RefreshToken, Project, Issue, Label, Comment. Миграция `init-all-entities` применена.

### Следующие шаги

- Issues — базовый flow. Создание / просмотр / обновление / удаление Issues внутри Project, статусы и переходы между ними.
- Issues — комментарии, фильтры, поиск. Полный пользовательский путь по бэклогу: комментарии, фильтрация по status / assignee / labels, поиск по тексту, пагинация.
- AI-триаж. Первая AI-фича. На вход — сам Issue (title + description) и контекст проекта (name, description, список доступных labels). На выход — priority, labels и черновик acceptance criteria. Записываем в Issue, показываем пользователю.
- AI-ревьюер PR. Вторая AI-фича, многошаговый агент с tool use. Пользователь даёт URL PR, агент через кастомные tools (fetch_pr_metadata, fetch_pr_diff, fetch_changed_files, fetch_file_content) сам решает, что подтянуть из репозитория и в каком порядке. Tools — обычные функции на бэкенде, которые ходят в GitHub REST API. Ревью оставляет комментарием к Issue.
- AI-фичи — крайние случаи и валидация. Что если модель вернула мусор. Что если diff пустой или огромный. Что если у проекта нет меток. Что если acceptance criteria противоречат. Защитный промптинг, повторы, явные fallback'и.
- Свой MCP-сервер наружу. Наш трекер выставляет MCP-сервер, чтобы разработчик из Claude Code мог напрямую работать с Issues и комментариями нашего трекера. Это не «подключи чужой MCP к нашему продукту», а противоположное — наш продукт интегрируется в чужие AI-агенты.
